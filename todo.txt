Verified & Improved Reconnection Flow Specifications:
--------------------------------------------------------------------------------
1. Room Cleanup for Solo Games:
   - If the roomId matches "SOLO-*" and the player disconnects, destroy the game state and the room state immediately in Redis. There is no need to keep bot-only matches running.

2. PostgreSQL Profile Mapping:
   - Add a column `active_room_id` to the users table.
   - When a match is started, write the roomId to `active_room_id` for all human players in that room.
   - When the match ends or the room is closed, clear the column (`SET active_room_id = NULL`).

3. Re-Authentication & Lobby Check:
   - On connecting to the site, check the database for `active_room_id`.
   - If present, fetch the game state from Redis:
     * If the state phase is "match_over" or "finished" (states 6, 7), display a locking overlay ("Your match has ended. Returning to lobby...") and clear the database column, then route to `/lobby`.
     * If phase is in-progress (state 4) and only 1 card remains in the trick/hand, show the locking overlay ("Trick is finishing...").
     * If the substitute bot is currently in the middle of executing a card play for them (during the 10s timeout or substitution delay), show a brief loading overlay until the bot completes the action, then render the game board.
     * Otherwise, route them directly to `/game/[id]` to seamlessly resume play.

4. Concurrent Sessions (Multi-tab):
   - When a user connects from a new socket, check if there is an existing socket connection for that userId.
   - If found, emit a "kick_concurrent" event to the old socket, force it to disconnect, and bind the new socket to the room immediately.

5. Solo Play hSet Bug Fix:
   - Resolved a Redis hSet TypeError where `updateRoomMetadata` was called with an object instead of positional parameters.
   - Refactored `updateRoomMetadata` to dynamically accept both object-style field updates and standard positional parameters.

6. Game Loop Crash Fix:
   - Fixed a game engine crash in `validator.js` (TypeError: cannot read properties of null (reading '1') at getSuit).
   - The crash occurred because when resetting `currentTrick` for consecutive tricks, the engine set `ledBy: trickWinner` before any card was played, making subsequent suit validations check a null card index.
   - Changed the engine to reset `ledBy` to `null` on new tricks, letting `playCard` assign it dynamically when the first card is led.
   - Added a Game Loop Simulation integration test to `engine.test.js` to ensure the bug is fully patched.

7. Light Theme & Scroll-Free Responsive Layout:
   - Refactored `/`, `/lobby`, `/room/[id]`, and `/game/[id]` to use a clean, high-contrast light theme.
   - Designed the Waiting Room and Game Board to be entirely scroll-free.
   - Restructured the game page into a three-column layout: ScoreCard, turn info, and leave button on the left panel; a perfectly round centered felt table in the center panel; and a decreased-width chat panel on the right panel.
   - Positioned player seats symmetrically around the circular table.
   - Placed the player's fanned cards in a dedicated, clean bottom bar.
   - Programmed the server to automatically flush all stale solo keys (`room:SOLO-*` and `game:SOLO-*`) from Redis on startup.

8. Trump Selection & Card Sorting:
   - Limited card exposure during `trump_selection` phase so players are only dealt/offered their first 5 cards (enforced securely in `getPlayerView` on the server and visually rendered in `TrumpSelector`).
   - Integrated automatic sorting of the player's hand by suit (Spades, Hearts, Diamonds, Clubs) and rank (descending, A down to 2) inside the frontend store state.
